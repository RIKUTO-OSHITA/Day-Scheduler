## Context

日次の業務スケジュールを時間軸で管理するため、Outlook 風の1日ビューを HTML で提供する。対象時間は 7:30-18:00 に固定され、タスク時刻は 15 分刻みで入力できる必要がある。既存システム連携やバックエンド永続化は要件に含まれず、単一ページのフロントエンド実装で完結する。

## Goals / Non-Goals

**Goals:**
- 1日スケジュールを縦方向の時間軸 UI で表示する。
- 7:30-18:00 の範囲外時刻が入力・表示されないようにする。
- タスクの開始時刻を 15 分刻みに正規化して入力できるようにする。
- 画面左上に表示対象日付を常時表示する。
- ブラウザで直接開ける HTML 出力を提供する。

**Non-Goals:**
- サーバーサイド API、認証、マルチユーザー共有。
- タイムゾーン変換や繰り返し予定の高度機能。
- モバイルアプリやデスクトップネイティブ化。

## Decisions

- 技術構成は HTML + CSS + JavaScript のシンプル構成とする。
理由: 要件が「出力は HTML」であり、導入コストと実装スピードを優先できる。
代替案: フレームワーク（React/Vue）導入は将来拡張性は高いが、現時点では過剰。

- レイアウトは左に時刻目盛り、右にタスク配置の2カラムとする。
理由: Outlook 風の視認性を再現しつつ、実装が単純。
代替案: 単一リスト表示は実装容易だが「Outlook みたいに」の要求を満たしにくい。

- 15分刻み制約は入力時にスナップ（丸め）し、表示も同じ粒度で扱う。
理由: 要件順守を UI 操作時点で担保でき、不正データを残さない。
代替案: バリデーションのみで拒否すると入力体験が悪化する。

- 日付表示はページ左上の固定ヘッダー要素で管理する。
理由: 常時視認が必要で、スクロール時にも位置を維持しやすい。
代替案: タイトル行のみは表示が埋もれる可能性がある。

## Risks / Trade-offs

- [Risk] 15分刻みの制約を見落として任意分入力される可能性 → Mitigation: 入力 UI で許容刻みのみ選択可能にし、ヘルプ文を表示する。
- [Risk] 7:30 開始の目盛り計算で表示ズレが起こる可能性 → Mitigation: 分単位からピクセル変換する共通関数を1箇所に集約する。
- [Trade-off] フレームワーク非採用により構造化や再利用性は限定される → Mitigation: データモデルと描画ロジックを関数分離し、将来移行しやすくする。
